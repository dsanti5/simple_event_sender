Testing

Intro

The tests are made to meet the specified requirements in the SRS, not in collaboration with developers. This is because the testing should be independently done with regards to the development of the product. Each requirement is broken down into test cases, where each test case is labeled with a test id. The test id:s for a specific requirement together validate the requirement. Most requirements are split into two test cases, one with positive formulation and one with negative formulation. Positive formulation means that a feature of the product regarding the requirement is tested with expected outcome. Negative formulation means that the test purposely uses the product incorrectly with expected error message. The tests are based on the framework Jest.

Structure and standards of the test files

In the repository for the project, there exists a map called “tests”. Each requirement has a file in this map called “FRXX.test.js”, where XX is the requirement number. Each file contains all test cases for respective requirement. In the test files, each test is named "Test id:YY for FRXX", where YY is the test id number and XX is the requirement number.

How to run tests

Single tests

In order to be able to run tests for individual requirement, following line needs to be run:

npm i -g jest-cli

Make sure that the server is up and running by executing the following line:

docker-compose up

Open a terminal in the “test”-folder and run:

jest FR[XX].test.js

Where [XX] is the requirement number. This will run the test cases for one test and output the test results, displaying any error codes.

Entire test suite

Make sure that the server is up and running by executing the following line:

docker-compose up

Open a terminal in the base of the project repo and run:

npm test

This will run the entire test suite (files in the repo with the ending “.test.js”) and output the test results, displaying error codes for the failed tests.

Continuous Integration

When a push to the repository is made your code is compiled and is run on the Gitlab CI/CD. The current CI settings and scripts are specified in the file .gitlab-ci.yml.

CI stages:

  • start: Runs the application for 30 secounds. Fails if the application crashes during this time.
  • test: Runs the tests specified in the /test directory. Fails if any test is failed. (Click on the job to view the console output and view which tests passed/failed)
